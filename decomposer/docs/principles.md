# Principles: how this workflow stays *generic*, *grounded*, and *auditable*

> This document is the constitution of the decomposer workflow. Every skill, every
> run artifact, and every design decision in this project is answerable to it.
> It exists to satisfy three demands from the project brief:
>
> 1. Each step must contain three phases — **论证 / 构建 / 独立检验**
>    (Argue / Build / Independently-Verify) — and all three must be grounded in the
>    *fundamental / motivational principles* of the task at hand.
> 2. We must have a *systematic, reusable* way to **extract those principles from a
>    new, possibly-unknown task** — so the workflow is not tied to one document
>    domain, format, or task type.
> 3. The end product feeds OpenPriors' ultimate goal: a formal
>    `(object, attribute, relation)` representation of concepts and reasoning chains.

---

## 0. Why a *decomposition* task even has "fundamental principles"

A naïve view says "parsing a document is just engineering — write a parser." That
view produces brittle, un-auditable, domain-locked code. The OpenPriors brief
instead asks for a workflow whose every move is *derived*, not *improvised*.

The escape from improvisation is to notice that **document decomposition is an
instance of a small number of task-independent invariants**. Whatever the document
(a LaTeX research paper, an exam problem with bespoke macros, a plain-prose riddle,
a metadata form), the same four questions have answers, and those answers *are* the
principles:

| Principle (the question) | Formal name | Measurable form (the lever for verification) |
|---|---|---|
| **P1. What must be conserved?** | *Faithfulness* / losslessness | Every input character is either (a) reproduced in some node's payload, or (b) explicitly classified as removable redundancy with a logged rule. `coverage = conserved_chars / total_chars` → target 1.0 minus logged redundancy. |
| **P2. What is redundant?** | *Minimum Description Length* (MDL) / Kolmogorov | A transform is justified iff it strictly shrinks the description **without** shrinking recoverable content: it must be *information-preserving up to a declared equivalence*. Measured by `len(before) − len(after)` paired with a round-trip equivalence check. |
| **P3. What types recur?** | *Compositionality* / type discovery | The document is generated by a small grammar of **data classes** (section, theorem-like block, equation, citation, …) that recur at many depths. Measured by: a finite class inventory covering ≥X% of blocks, with an "unclassified" residual that must be small and itemized. |
| **P4. What is the nesting & reference structure?** | *Hierarchy + referential integrity* | The classified blocks form a tree by containment **and** a graph by cross-reference (`\ref`, `\cite`, `\eqref`). Measured by: tree is well-formed (single root, no orphans) and `dangling_refs = 0` (every reference resolves to a node, or is logged as external). |

These four are not specific to LaTeX or to math. They are the conserved quantities
of *any* "raw artifact → clean structured representation" task. That is precisely
why the workflow built on them generalizes. **Section 4 gives the procedure for
re-deriving P1–P4 (and any task-specific P5…) for an arbitrary new corpus.**

### The bridge to OpenPriors' end goal

The end goal is an `(object, attribute, relation)` graph of concepts and reasoning.
Our four principles map onto it directly, which is why this decomposer is the
*correct* upstream stage and not an arbitrary preprocessing choice:

- **P3 (recurring types)** → the **node/edge *type system*** of the eventual graph.
  A `Theorem` block is a node type; a `cites` edge is a relation type.
- **P4 (hierarchy + refs)** → the graph's **edges**: `contains` (tree) and
  `references` (cross-ref) are the first two relations; later stages add semantic
  relations (`proves`, `assumes`, `generalizes`) on the same substrate.
- **P1 (faithfulness)** guarantees the graph is *about the actual document* and not
  a hallucinated paraphrase — the property a structure-mapping / novelty detector
  most needs (see `Imports/structure mapping notes.md` §1: "isomorphism under
  renaming, not lexical overlap" requires a *faithful* source representation).
- **P2 (MDL)** is the same objective the novelty detector uses downstream
  ("minimal generating backbone", structure-mapping notes §6); removing surface
  redundancy *now* is the first increment of the description-length minimization
  the whole project is built around.

## 1. The three-phase doctrine: Argue / Build / Verify (论证 / 构建 / 独立检验)

Every step in the workflow — at the granularity of a *skill* — produces a triad of
artifacts. The triad is mandatory; a step with a missing phase is not "done".

### Phase A — 论证 / Argue (`rationale.md`)

*Derive the step from P1–P4 before writing any code.* The rationale must state:

- **Which principles this step serves** and the **measurable target** it adopts
  from the table above.
- A **literature/tooling review** (`docs/lit-review/*`): what already exists, what
  we adopt, what we reject and *why* (去伪存真 — separate the real from the
  fake/over-hyped). No tool is used without a recorded comparison.
- The **decision** and its **performance-vs-resource** justification, recorded as an
  ADR in `docs/decisions/`.

### Phase B — 构建 / Build (the transform + its output)

*The smallest deterministic transform that meets the target.* Build doctrine:

- **Deterministic-by-default.** Prefer code with reproducible output over an LLM
  call. LLMs are used **only** where genuine natural-language *understanding* is
  irreducible (e.g. discovering data classes in an unfamiliar format, judging
  semantic redundancy). This is a *performance, cost, and auditability* decision,
  not a stylistic one: deterministic transforms are free, fast, and diffable.
- **Sub-agents, not API calls.** Where NL understanding is needed, it is delegated
  to in-session Claude Code **sub-agents** (covered by the Max plan), never to a
  metered API. Their prompts and outputs are saved as run artifacts.
- **Canonical, diffable output.** All structured output is emitted with sorted keys,
  stable content-addressed IDs, and `\n`-terminated lines so that re-runs produce
  minimal git diffs (auditability).

### Phase C — 独立检验 / Independently Verify (`verification.md` + checks)

*A check that does **not** share the build's failure modes.* If the build and the
check can fail together for the same reason, the check is theater. Independence is
engineered via the menu in §3. Each step declares, in `verification.md`:

- the **invariants** it asserts (from P1–P4), each as a machine-checkable predicate;
- the **independence argument**: why a bug in the build would *not* also fool the
  check (e.g. the build expands macros by string rewriting; the check confirms
  losslessness by an *independent* character-coverage accounting + a structural
  re-parse, and an LLM sub-agent that never saw the build's code spot-reads the
  output against the raw source);
- the **measured results** of the checks on every corpus item, with pass/fail.

## 2. First-principles objectives, in measurable form

The principles of §0 become concrete objectives the build optimizes and the
verifier measures. (Fuller literature in `docs/lit-review/03-verification-and-principles.md`.)

- **Faithfulness (P1) ⇒ lossless round-trip / conservation.** The strongest form:
  an inverse transform reconstructs the input exactly. The weakest usable form:
  *character-coverage conservation* — every non-redundant input character is
  attributable to exactly one leaf node. We use conservation as the always-on
  invariant and round-trip where an inverse is cheap.
- **MDL / redundancy (P2) ⇒ shrink-with-equivalence.** A normalization step is
  *valid* iff `description_length` strictly decreases **and** an equivalence oracle
  certifies no content was lost (e.g. comment stripping: the stripped text must be
  byte-identical after re-inserting comments as no-ops; macro expansion: the
  detokenized meaning is unchanged). Redundancy we *remove* is always *logged*, so
  removal is reversible on paper.
- **Compositionality (P3) ⇒ grammar coverage.** Measured as the fraction of source
  blocks assigned to a named class, with the unclassified residual enumerated. A
  good class inventory is **small** (few types) yet **high-coverage** (most blocks)
  — itself an MDL criterion on the *schema*.
- **Referential integrity (P4) ⇒ zero dangling edges.** Every `\label` is a node id;
  every `\ref/\eqref/\cite` is an edge to such an id or is logged as external. The
  containment tree must have exactly one root and no cycles.

The lexical < conceptual < logical hierarchy that OpenPriors cares about
(`Imports/Research Report.md`) is honoured here at the *lexical/structural* floor:
this workflow nails the surface/structural layer **faithfully** so downstream
stages can lift it to the conceptual and logical layers without inheriting surface
noise.

## 3. The independent-verification menu (engineering "检验" so it is real)

A build and its verifier are *independent* when they rest on different mechanisms.
Techniques, with how each is instantiated here:

1. **Conservation / accounting check** — count a conserved quantity (characters,
   tokens, blocks) before and after; independent because it ignores *how* the
   transform works and only audits totals.
2. **Round-trip / inverse testing** — `parse → serialize → re-parse` must be a
   fixed point; independent because the serializer and parser are written against
   the schema, not against each other's internals.
3. **Differential testing** — two independent extractors (e.g. our LaTeX walker vs.
   the ar5iv HTML structure for the *same* paper) must agree on the section/theorem
   skeleton; disagreement localizes a bug.
4. **Metamorphic testing** — apply a meaning-preserving mutation to the input
   (rename a macro, reorder comments, renumber a label) and assert the structured
   output changes only in the predicted way. Directly tests the renaming-invariance
   OpenPriors needs.
5. **Schema validation** — every emitted node is validated against the Pydantic
   schema / emitted JSON Schema; independent because the schema is a separate
   declarative artifact.
6. **Idempotence** — normalization applied twice equals applied once; cheap, strong.
7. **Cross-reader LLM audit** — a sub-agent that *did not* see the build code reads
   the raw source and the structured output and reports mismatches in natural
   language. Independent because its mechanism (reading comprehension) shares no
   code with the deterministic build. Used as a *spot-check on top of* the
   deterministic invariants, never as the sole gate (LLM-as-judge is prompt- and
   conformity-biased; it complements, it does not replace, the machine checks).

A step picks the **subset of these whose joint failure with the build is least
plausible**, and argues that choice in `verification.md`.

## 4. The reusable procedure: extracting principles from an *unknown* task

This is the part that makes the workflow generic rather than a one-off for these
four documents. Given a new corpus of unknown domain/format/task, run this
**Principle-Extraction Protocol (PEP)** before designing any step:

> **PEP-1. Name the conserved quantity.** Ask "if I transform this artifact, what
> must survive unchanged for the output to still be *about* the input?" The answer
> instantiates **P1** (here: characters/tokens of the source). Write the
> conservation predicate.
>
> **PEP-2. Name the redundancy.** Ask "what in this format carries no
> information about content — only about presentation or the tooling?" (LaTeX
> comments, whitespace, presentation macros, boilerplate). Each becomes a logged,
> reversible removal rule under **P2**. The test: removal must pass PEP-1's
> conservation up to the declared equivalence.
>
> **PEP-3. Discover the recurring types (the data classes).** Sample the corpus and
> ask "which structural units appear many times and play a *segmenting* role?"
> Cluster by *role*, not by surface keyword, so the inventory transfers across
> formats (a `theorem` in `\begin{theorem}`, in a `\newtheorem`-defined env, in a
> custom `msbproblem`, and in a bolded prose "Theorem 1." are *the same class*).
> This is the step that most needs an LLM sub-agent, because "same role, different
> surface" is a natural-language judgment — and it is exactly the structure-mapping
> intuition (relations over surface). Output: a class inventory with evidence.
>
> **PEP-4. Recover hierarchy and references.** Ask "what contains what (tree) and
> what points to what (graph)?" Define the containment relation and the reference
> relations. Instantiate **P4**.
>
> **PEP-5. Look for task-specific principles P5…** Some corpora add invariants
> (e.g. an exam problem must keep *statement* and *answer key* separable; a dataset
> form must preserve field identity). Add them only when a conserved quantity or a
> required equivalence is found that P1–P4 do not already cover.

Each PEP question yields (a) a principle, (b) a measurable target, and therefore
(c) a verifier. The Argue phase of every downstream step then *cites which PEP
output it serves*. The protocol is logged per corpus in
`runs/<doc>/00-principle-extraction/`.

**Why role-clustering generalizes (the core bet).** Surface markup is unbounded and
idiosyncratic (every LaTeX preamble invents macros; every house style renames
environments). *Roles* are few and stable across human technical writing:
**segmenting** (sections), **asserting** (theorems/definitions/claims),
**formalizing** (equations/displayed math), **referring** (citations/cross-refs),
**exemplifying** (examples/problems), **meta** (comments/notes/grading). A workflow
keyed to roles, with format-specific *adapters* feeding a format-agnostic *core*,
is the design that survives an unknown next document. This is the same move
structure-mapping theory makes — and the reason OpenPriors' own notes are the
intellectual parent of this file.

## 5. How the artifacts on disk enforce all of the above

- `skills/_template/SKILL.md` bakes the A/B/C triad into every skill.
- `runs/<doc>/<NN-step>/{rationale.md, <output>, verification.md, metrics.json}`
  is the uniform, informative per-step record — full metadata + the development
  trajectory — required by the brief.
- `docs/lit-review/*` holds the "吸收学习对比反思" reviews; `docs/decisions/*` holds
  the ADRs with the performance-vs-resource argument; `docs/tool-guide.md` is the
  distilled, reusable guide.
- `src/decomposer/schema.py` is the single source of truth for the data classes
  (P3) and emits JSON Schema for audit.
- `src/decomposer/verify/` holds the reusable invariant checkers (§3) so that
  "独立检验" is library code, not prose.
